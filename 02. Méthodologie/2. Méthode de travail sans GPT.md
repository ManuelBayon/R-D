## 0. Clarifier le risque avant tout

Avant même la doc :

- quel problème je résous ?  
- quel bug / risque je veux empêcher ?  
- quelle couche consommera ce composant ?

👉 sans ça, tu écris dans le vide

---

## 1. Contrat court (pas doc longue)

- responsabilité  
- input / output  
- ownership  
- 2–3 invariants max  
- hors périmètre

👉 une page, pas plus

---
## 2. Diagramme si et seulement si il réduit l’ambiguïté

Pas un réflexe automatique.

À faire seulement si ça aide à voir :

- flux
- ownership
- dépendances
- séparation état / vue

---
## 3. Implémentation minimale qui peut échouer pour de vrai

Très important.

Pas :

```
code complet
```

Mais :

```
plus petite version exécutable qui révèle les vrais problèmes
```

---
## 4. Tests orientés contrat

Pas “je teste tout”.  
Tu testes :

- invariants critiques
- cas limites structurants
- comportement d’erreur

---
## 5. Faire tourner et observer

Un mentor insisterait sur ça :

- où ça frotte ?  
- qu’est-ce qui est pénible ?  
- qu’est-ce qui duplique ?  
- qu’est-ce qui est mal placé ?

👉 le design se révèle dans l’usage

---
## 6. Self-review écrite

Très bon point dans ta version.  
À garder absolument.

Format conseillé :

- ce qui est cohérent  
- ce qui est flou  
- ce qui est fragile  
- ce que je déplacerais

---
## 7. Refactor ciblé

Uniquement si le code ou les tests t’ont appris quelque chose.

Pas de “refactor esthétique”.

---

## 8. Review externe

Oui. Mais une review utile doit être formulée comme ça :

```
voici mes choix  
voici mes doutes  
challenge-moi sur responsabilités / invariants / couplage
```

Pas :

```
est-ce que c’est bien ?
```

---
## 9. Itération ou freeze

Très important.

Après review, tu dois décider :

```
- est-ce que je freeze et j’avance ?  
- ou est-ce qu’il y a un problème structurel réel à corriger ?
```
👉 sinon tu peux itérer sans fin

---
# Donc la boucle mentor top-tier serait plutôt :

0. risque / problème à résoudre  
1. contrat court  
2. diagramme si utile  
3. code minimal exécutable  
4. tests de contrat  
5. observation réelle  
6. self-review écrite  
7. refactor ciblé  
8. review externe  
9. freeze ou itération
