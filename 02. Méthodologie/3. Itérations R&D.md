```
1. définir une propriété/invariant
2. formaliser minimalement (mini spec)
3. écrire un test qui peut casser
4. implémenter le minimum
5. vérifier / casser / corriger
6. verrouiller
```

---
# Exemple concret

Mauvaise itération :

```
“je code un journal”
→ test
→ ça marche
```

Bonne itération :

```
propriété :
“le journal est append-only et ordonné”

test :
→ tentative de mutation
→ tentative d’insertion hors ordre

implémentation :
→ minimum pour satisfaire ces contraintes
```

---
# Boucle à suivre

```
1. formuler l’invariant en une phrase testable
2. écrire 2–3 tests qui tentent de le casser
3. implémenter le minimum
4. vérifier que les tests passent
5. essayer mentalement de le casser autrement
6. ajouter un test si nécessaire
7. verrouiller
```

---
# Pourquoi ?

Parce que :

```
les bugs viennent des propriétés violées
pas du code en lui-même
```

---
# Boucle mentale

```
“qu’est-ce qui doit toujours être vrai ?”
→ “comment je le casse ?”
→ “comment je l’empêche ?”
```

---
# A éviter absolument

```
- coder longtemps sans test
- design complet upfront
- refactor sans invariant
- ajouter des features sans preuve
```
